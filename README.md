# Hustad Residential Tablet Platform

Tablet-first storm inspection-to-sale workflow for Hustad residential reps.  
**Stack:** Next.js 14 · TypeScript · Tailwind CSS · Supabase (Postgres + Storage) · Render

---

## Quick Start (Local Development)

```bash
git clone <your-repo-url>
cd hustad-tablet-platform
cp .env.local.example .env.local  # Fill in your Supabase credentials
npm install
npm run dev
```

Open [http://localhost:3000](http://localhost:3000)

## Deployment Guide

### Step 1: Create Supabase Project

1. Go to [supabase.com](https://supabase.com) → New Project
2. Choose a name (e.g. `hustad-tablet`) and region close to Madison, WI
3. Save your database password
4. Once created, go to **Settings → API** and copy:
   - `Project URL` → this is your `NEXT_PUBLIC_SUPABASE_URL`
   - `anon public` key → this is your `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - `service_role` key → this is your `SUPABASE_SERVICE_ROLE_KEY`

### Step 2: Run Database Migration

1. In Supabase Dashboard, go to **SQL Editor**
2. Click **New Query**
3. Paste the contents of `supabase/migrations/001_initial_schema.sql`
4. Click **Run** — this creates all tables, indexes, and RLS policies

### Step 3: Create Storage Bucket

1. In Supabase Dashboard, go to **Storage**
2. Click **New Bucket**
3. Name: `inspection-photos`
4. Set to **Private** (photos accessed via signed URLs)

### Step 4: Seed Default Users

```bash
# With .env.local filled in:
npm install
node scripts/seed.js
```

This creates:
- Admin: `admin@hustad.com` / password: `hustad2026` / PIN: `1234`
- Rep: `rep@hustad.com` / password: `field2026` / PIN: `5678`

**⚠️ Change these passwords before going live.**

### Step 5: Deploy to Render

1. Push this repo to GitHub
2. Go to [render.com](https://render.com) → New → **Web Service**
3. Connect your GitHub repo
4. Render will detect `render.yaml` automatically
5. Set environment variables in Render dashboard:
   - `NEXT_PUBLIC_SUPABASE_URL` — your Supabase project URL
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY` — your anon key
   - `SUPABASE_SERVICE_ROLE_KEY` — your service role key
   - `JWT_SECRET` — auto-generated by Render, or set your own 64-char hex
   - `NEXT_PUBLIC_APP_URL` — your Render URL (e.g. `https://hustad-tablet.onrender.com`)
6. Click **Deploy**

Build command: `npm install && npm run build`  
Start command: `npm start`

```
│   ├── lib/
│   │   ├── session.ts                 # Session factory + localStorage persistence
│   │   ├── auth.ts                    # JWT sign/verify, bcrypt helpers
│   │   ├── supabase.ts               # Browser Supabase client

```

## How Data Flows

```
iPad (field)
  ├── localStorage (instant save, survives offline)
  ├── SessionProvider (React state machine)
  └── sync.ts (debounced 2s)
        ├── ONLINE → POST /api/sessions → Supabase
        └── OFFLINE → localStorage sync queue
                        └── processQueue() on reconnect → Supabase
```

## API Endpoints

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| POST | `/api/auth` | none | Login (email + password/PIN) |
| GET | `/api/auth` | token | Check session |
| DELETE | `/api/auth` | token | Logout |
| GET | `/api/sessions` | token | List rep's sessions |
| POST | `/api/sessions` | token | Create or sync session (upsert) |
| GET | `/api/sessions/[id]` | token | Get session + photos + tasks + audit |
| DELETE | `/api/sessions/[id]` | token | Archive session |
| POST | `/api/photos` | token | Upload photo to Supabase Storage |
| GET | `/api/export?session_id=x` | token | Download session as JSON |

## Database Tables

| Table | Purpose |
|-------|---------|
| `reps` | Rep accounts with bcrypt passwords and PINs |
| `sessions` | Full session state (mirrors frontend SessionState) |
| `photo_assets` | Photo metadata + Supabase Storage paths |
| `follow_up_tasks` | Deferred/absent DM follow-up tasks |
| `audit_events` | Complete audit log of all state changes |

## Security

- JWT tokens in httpOnly cookies (12h expiry)
- Supabase Row Level Security enabled on all tables
- Service role key used only in server-side API routes
- Passwords and PINs bcrypt hashed
- No PII in client-side analytics
